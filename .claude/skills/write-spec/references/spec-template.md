## 시나리오

시나리오마다 `S1`, `S2` 형식의 ID를 붙인다. Then이 곧 판정 기준 체크리스트다.
체크박스는 실행 증거(테스트 통과 또는 End-to-end 확인)로만 켠다.

### S1. 시나리오 제목
- **Given**: 사용자나 시스템의 초기 상태
- **When**: 사용자가 취하는 행동
- **Then**
  - [ ] S1-1 구체적 입력 → 관찰 가능한 결과 (UI, API 응답, 저장 상태)
  - [ ] S1-2 구체적 입력 → 관찰 가능한 결과

### S2. 시나리오 제목
- **Given**: ...
- **When**: ...
- **Then**
  - [ ] 관찰 가능한 결과 (판정 기준이 하나면 세부 ID를 생략하고 `S2`로 참조한다)

happy path와 변경 비용이 큰 error branch를 모두 덮을 만큼 시나리오를 만든다.

## 불변 규칙 (선택)

# 작성 가이드

## 판정 기준의 시험: "이 한 줄만 보고 테스트를 쓸 수 있는가?"

No라면 그 항목이 너무 모호하거나, 관찰 가능한 행동이 아니라 구현을 기술하고 있다.

## ID 규칙

ID는 plan과 테스트가 참조하는 앵커다. ID가 흔들리면 참조 전체가 깨진다.

- 시나리오 ID(`S1`)는 필수다. Then 항목이 둘 이상이면 기준 ID(`S1-1`)를 붙이고, 하나면 시나리오 ID가 그 기준의 ID를 겸한다.
- ID는 추가만 한다. 재배열과 재사용을 금지한다.
- 기준을 삭제할 때는 번호를 결번으로 남긴다: `~~S1-2~~ (삭제: 사유)`
- 기준의 의미가 바뀌면 새 ID를 발급한다. 같은 번호에 다른 뜻을 담지 않는다.

## 각 필드에 무엇을 쓰는가

| 필드 | 무엇을 쓰는가 | 허용 | 금지 |
|---|---|---|---|
| **Given** | 관찰 가능한 선행 조건: 화면 상태, 데이터 개수, 적용된 필터 | "할 일 3개가 표시된 리스트", "검색창에 'milk' 입력됨" | "todos.length === 3", "isLoading is true" |
| **When** | 구체적인 UI 요소와 그 위의 행동 | "'삭제' 버튼 클릭", "제목 필드에 'Meeting notes' 입력" | "deleteTodo() 호출", "SUBMIT 액션 디스패치" |
| **Then** | 컴포넌트 바깥에서 단언 가능한 결과의 체크리스트 | "'할 일 2개' 텍스트가 표시된다" | "todos.length === 2", "API가 호출된다" |

## Then 항목(판정 기준) 규칙

각 항목은 **입력 → 관찰 가능한 출력** 한 쌍이고, 테스트 하나에 매핑된다.

### 허용되는 출력

| 유형 | 예시 |
|---|---|
| 보이는 텍스트 | `"항목 추가됨"`이 나타난다 |
| 요소 개수 | 카드 3개가 렌더된다 |
| 요소 존재 | "삭제" 버튼이 보인다 |
| 필드 값 | 제목 필드에 `"우유 사기"` |
| 시각 상태 (클래스/속성으로 표현 가능) | 완료된 항목에 취소선 |
| API 응답 형태 (feature가 API를 노출할 때) | `200 { id: <new-id> }` |

### 금지되는 출력

| 유형 | 이유 |
|---|---|
| 내부 상태 (store, React state, signals) | 사용자가 관찰할 수 없다 |
| 함수 호출 단언 (`mockFn was called`) | 구현에 결합된다. 리팩터링하면 깨진다 |
| 원시 DB 행, 서비스 계층 반환값 | 서버 내부라 사용자가 볼 수 없다 |

카피 문구처럼 바꾸기 쉬운 값에는 `(제안 기본값)`을 붙인다. 문구를 다듬는 일이 spec 개정으로 번지지 않게 한다.

### 좋은 예 vs 나쁜 예

```
좋음:
  - [ ] S1-1 title="우유 사기" → 텍스트 "우유 사기"인 항목이 추가되고 리스트가 3개가 된다
  - [ ] S1-2 빈 제목 → 필드 아래 "제목을 입력하세요"(제안 기본값)가 나타난다

나쁨:
  - [ ] todos 배열이 [{ id: 1, title: "우유 사기", done: false }]가 된다   ← 내부 상태
  - [ ] addTodo()가 { title: "우유 사기" }로 한 번 호출된다                ← 구현 결합
```

나쁜 버전으로 쓴 테스트는 컴포넌트 내부를 mock해야 하고, 구현을 바꾸면 깨지고, 사용자 행동에 대해 아무것도 증명하지 않는다.

## 불변 규칙을 쓸 때

3개 이상 시나리오에 거의 동일한 Then 항목을 쓰고 있다면 그 규칙은 불변 규칙이다. 모든 경로에 적용되므로 단일 시나리오에 담을 수 없다.

| 범주 | 예시 |
|---|---|
| **보안 / 프라이버시** | 다른 사용자의 비공개 드래프트는 어떤 경로(UI, URL, 검색, RSS)로도 보이지 않는다 |
| **성능** | 모든 페이지가 4G에서 2초 안에 first contentful paint에 도달한다 |
| **데이터 일관성** | 표시되는 업보트 수는 기록된 투표 수와 항상 일치한다 |

## End-to-end 검증을 쓸 때

- 사용자 입장에서 feature를 처음부터 끝까지 통과하는 절차 하나를 쓴다
- 각 단계가 커버하는 ID를 표기해서, 절차가 spec 전체를 훑는지 눈으로 확인할 수 있게 한다
- 실행 도구(Playwright, Browser MCP, 수동 확인)의 선택은 plan.md의 몫이다
